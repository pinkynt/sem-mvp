# Mini manual de identidad visual — SEM Digital / Municipalidad de Salta

## 1. Objetivo del manual

Este documento define lineamientos visuales para diseñar la interfaz del prototipo SEM Digital manteniendo coherencia con la identidad visual utilizada actualmente por la Municipalidad de Salta.

No reemplaza un manual oficial de marca. Es una guía práctica para hackaton, construida a partir del análisis visual de la web municipal, piezas gráficas institucionales y materiales digitales de la Municipalidad.

El objetivo es que el sistema se perciba como:

- Institucional.
- Simple.
- Moderno.
- Accesible.
- Confiable.
- Integrado al ecosistema digital municipal.

---

## 2. Personalidad visual de la marca

La identidad de la Municipalidad de Salta transmite una imagen principalmente:

- Institucional.
- Digital.
- Clara.
- Cercana.
- De servicio público.
- Moderna, pero sin perder formalidad.

La comunicación visual no es solemne ni burocrática. Se apoya en bloques grandes, colores planos, tipografía redondeada y mensajes directos.

Para SEM Digital, la interfaz debería verse como una extensión natural de los servicios digitales municipales, no como una app privada ni como una fintech.

---

## 3. Concepto visual general

La línea gráfica observada se organiza en torno a:

- Fondos azules institucionales.
- Cards grandes y rectangulares con bordes redondeados.
- Tipografía blanca sobre azul.
- Íconos grandes, lineales y de fácil lectura.
- Uso de formas geométricas de fondo en baja opacidad.
- Contrastes fuertes.
- Mensajes breves.
- Botones tipo píldora.
- Fotografías recortadas dentro de formas curvas o geométricas.

Para SEM Digital, la recomendación es mantener una estética de servicio municipal digital, con una interfaz más funcional y simple que publicitaria.

---

## 4. Paleta cromática recomendada

### Color primario — Azul Municipalidad

Uso principal:

- Header.
- Botones principales.
- Fondos destacados.
- Cards institucionales.
- Pantallas de inicio.

Valor aproximado:

```text
Azul primario: #005CB9 / #0066C9
```

Este color debe ser el protagonista del sistema.

---

### Azul oscuro institucional

Uso principal:

- Fondos de alto contraste.
- Secciones importantes.
- Sidebar o encabezados.
- Pantallas de confirmación más sobrias.
- Elementos de jerarquía.

Valor aproximado:

```text
Azul oscuro: #003B66 / #00416D
```

---

### Celeste / azul secundario

Uso principal:

- Fondos decorativos.
- Hover states.
- Indicadores suaves.
- Detalles de interfaz.
- Degradados.

Valor aproximado:

```text
Celeste institucional: #00A8D8 / #00BBD4
```

---

### Verde de confirmación

Uso principal:

- Pago confirmado.
- Estados correctos.
- Tildes.
- Mensajes positivos.

Valor aproximado:

```text
Verde confirmación: #7ED957 / #8BD986
```

En SEM Digital, este color debe usarse con mucha intención. Por ejemplo:

```text
✅ Pago confirmado
```

---

### Amarillo de apoyo

Uso principal:

- Íconos secundarios.
- Elementos destacados.
- Advertencias suaves.
- Información importante no crítica.

Valor aproximado:

```text
Amarillo apoyo: #FFC928 / #FFD23F
```

No debe competir con el azul. Usarlo de forma puntual.

---

### Blanco

Uso principal:

- Fondos.
- Cards.
- Texto sobre azul.
- Espacios de descanso visual.

Valor:

```text
Blanco: #FFFFFF
```

---

### Gris claro

Uso principal:

- Fondos secundarios.
- Separadores.
- Estados inactivos.
- Contenedores suaves.

Valor aproximado:

```text
Gris claro: #F2F5F8 / #EEF2F5
```

---

## 5. Uso del color en SEM Digital

### Botón principal

Usar azul primario con texto blanco.

Ejemplo:

```text
[NUEVO COBRO]
```

---

### Confirmación de pago

Usar fondo claro o blanco, tilde verde grande y texto azul oscuro.

Ejemplo:

```text
✅ Pago confirmado

Patente
AB 123 CD
```

---

### Advertencias

Usar amarillo para alertas preventivas.

Ejemplo:

```text
Horario próximo a finalizar
```

---

### Errores o bloqueos

Usar rojo solo cuando sea estrictamente necesario.

Ejemplo:

```text
No se puede cobrar en este horario
```

El rojo no aparece como color fuerte de la identidad observada, por lo tanto debe usarse con moderación.

---

## 6. Tipografía

La marca utiliza una tipografía de apariencia geométrica, redondeada, limpia y amable.

Para el prototipo se recomienda usar:

### Opción recomendada

```text
Montserrat
```

Por qué:

- Es legible.
- Es moderna.
- Tiene buen peso visual.
- Funciona bien en interfaces.
- Se parece al estilo institucional observado.

---

### Alternativas válidas

```text
Poppins
Nunito Sans
Inter
```

Para mantener una apariencia más cercana a las piezas municipales, Montserrat o Poppins serían las mejores opciones.

---

## 7. Jerarquía tipográfica

### Títulos principales

Uso:

- Pantallas principales.
- Cards destacadas.
- Confirmaciones.

Estilo:

```text
Peso: 700 / 800
Tamaño mobile: 28px - 36px
Color: azul oscuro o blanco
```

Ejemplo:

```text
Nuevo cobro
```

---

### Títulos secundarios

Uso:

- Subtítulos.
- Nombres de secciones.
- Estados.

Estilo:

```text
Peso: 600 / 700
Tamaño mobile: 20px - 24px
```

---

### Texto común

Uso:

- Indicaciones simples.
- Descripciones breves.

Estilo:

```text
Peso: 400 / 500
Tamaño mobile: 16px - 18px
```

---

### Texto para botones

Uso:

- Acciones principales.
- Opciones de tiempo.
- Confirmaciones.

Estilo:

```text
Peso: 700
Mayúsculas en acciones principales
Tamaño mínimo: 18px
```

Ejemplo:

```text
COBRÉ EN EFECTIVO
```

## 8. Logo y marca

Para el prototipo, se debe utilizar el logo de Salta Municipalidad en:

- Pantalla de inicio.
- Header del panel municipal.
- Comprobantes digitales.
- Tickets o representación de ticket.
- Pantalla de confirmación.

Recomendaciones:

- Usar versión blanca sobre fondo azul.
- Usar versión azul sobre fondo blanco.
- No deformar el logo.
- No aplicar sombras innecesarias.
- No cambiar colores del logo.
- Mantener margen de seguridad alrededor.

---

## 9. Formas y recursos gráficos

La identidad municipal utiliza formas geométricas suaves como fondo, generalmente en baja opacidad.

Recursos recomendados:

- Grandes formas curvas.
- Íconos lineales.
- Fondos con marca de agua.
- Patrones suaves derivados del isotipo.
- Rectángulos con bordes redondeados.
- Cards tipo banner.

Para SEM Digital, estos recursos deben usarse de manera moderada, porque la prioridad es la usabilidad.

---

## 10. Bordes y radios

Las piezas gráficas municipales utilizan esquinas redondeadas.

Recomendación para UI:

```text
Cards: 16px - 24px
Botones: 16px - 999px
Inputs: 12px - 16px
Modales: 20px - 24px
```

Los botones tipo píldora funcionan bien con la identidad observada.

---

## 11. Iconografía

Usar íconos simples, lineales y de alto contraste.

Estilo recomendado:

- Trazo grueso.
- Bordes redondeados.
- Pocas líneas.
- Fácil reconocimiento.
- Blanco sobre azul o azul sobre blanco.

Íconos útiles para SEM Digital:

- Auto.
- Moto.
- Ticket.
- QR.
- Moneda / efectivo.
- Tarjeta.
- Check.
- Reloj.
- Ubicación.
- Patente.
- Usuario / permisionario.
- Celular.
- POS.

Evitar íconos demasiado técnicos o abstractos.

---

## 12. Estilo de botones

### Botón principal

Uso:

- Nuevo cobro.
- Continuar.
- Confirmar.
- Cobré en efectivo.

Estilo:

```text
Fondo: azul primario
Texto: blanco
Borde: ninguno
Radio: píldora o 16px
Altura mínima mobile: 56px
Peso tipográfico: 700
```

Ejemplo:

```text
[NUEVO COBRO]
```

---

### Botón de confirmación positiva

Uso:

- Pago confirmado.
- Finalizar.
- Nuevo cobro después de confirmar.

Estilo:

```text
Fondo: verde confirmación
Texto: azul oscuro o blanco
```

---

### Botón secundario

Uso:

- Ver cobros.
- Ayuda.
- Volver.

Estilo:

```text
Fondo: blanco
Texto: azul primario
Borde: 1px sólido azul claro
```

---

### Botón peligroso

Uso:

- Cancelar cobro.
- Anular.
- Eliminar.

Recomendación:

No usarlo en pantallas principales del permisionario.

Si aparece, debe estar separado de la acción principal.

---

## 13. Inputs

Los campos deben ser grandes y muy legibles.

Para la patente:

```text
Altura mínima: 64px
Texto: 28px - 36px
Alineación: centrada
Formato visual: AB 123 CD
```

El input de patente debe tener ejemplo visible:

```text
Ejemplo: AB123CD
```

No pedir datos innecesarios.

---

## 14. Cards

Las cards deben ser simples y de lectura rápida.

Estilo:

```text
Fondo: blanco o azul
Radio: 20px
Padding: 20px - 28px
Sombra: suave o inexistente
```

Para el panel municipal, se pueden usar cards de métricas:

```text
Pagos hoy
$ Recaudación
Efectivo
QR Mercado Pago
```

Para la app del permisionario, evitar exceso de cards. Priorizar botones grandes.

---

## 15. Pantalla del permisionario

La pantalla del permisionario debe ser mucho más simple que el panel municipal.

No debe parecer un dashboard administrativo.

Debe parecer una herramienta de cobro rápida.

### Pantalla inicial sugerida

```text
Hola, Juan

Zona: Centro A

[NUEVO COBRO]

Cobros de hoy: 12
Total de hoy: $8.400

[Ver cobros]
[Ayuda]
```

Lineamientos:

- Un solo botón protagonista.
- Texto grande.
- Mucho aire.
- Pocas opciones.
- Sin tablas.
- Sin menú lateral.
- Sin información excesiva.

---

## 16. Flujo visual de cobro

El flujo debe ser:

```text
Nuevo cobro
→ Patente
→ Tiempo
→ Forma de pago
→ Confirmar
→ Pago confirmado
```

Cada paso debe ocupar una pantalla.

---

## 17. Pantalla de pago confirmado

Esta pantalla es crítica porque funciona como comprobante visual para el conductor cuando el permisionario usa smartphone.

Debe ser muy clara.

### Composición recomendada

```text
✅

Pago confirmado

Patente
AB 123 CD

Válido hasta
11:45

Pago
Efectivo

Código
SEM-48291

[NUEVO COBRO]
```

Lineamientos:

- Tilde verde grande.
- Patente grande.
- Hora de vencimiento grande.
- Código visible pero secundario.
- No colocar muchas acciones.
- Debe poder mostrarse al conductor desde cierta distancia.

---

## 18. Ticket POS

El ticket debe ser simple, institucional y directo.

Contenido recomendado:

```text
SEM DIGITAL
Municipalidad de Salta

PAGO CONFIRMADO

Patente: AB 123 CD
Tiempo: 1 hora
Válido hasta: 11:45
Pago: Efectivo
Código: SEM-48291

Conserve este comprobante
```

Lineamientos:

- Sin diseño complejo.
- Mucho contraste.
- Texto claro.
- Código visible.
- Logo o nombre municipal arriba.

---

### Pantalla de dashboard

```text
Pagos registrados hoy
Recaudación registrada
Pagos en efectivo
Pagos digitales
Permisionarios activos
Zonas activas
```

### Tabla de pagos

Columnas recomendadas:

## 21. Tono de comunicación

El tono debe ser:

- Claro.
- Directo.
- Amable.
- Institucional.
- Sin tecnicismos.
- Sin exceso de texto.

Ejemplos recomendados:

```text
Ingresá la patente
```

```text
Elegí el tiempo
```

```text
¿Cómo paga?
```

```text
Cobré en efectivo
```

```text
Pago confirmado
```

```text
No se puede cobrar ahora
```

Evitar:

```text
Registrar operación tarifaria
```

```text
Validar transacción
```

```text
Sincronizar comprobante
```

---

## 22. Accesibilidad

La interfaz debe contemplar usuarios con bajo nivel de alfabetización digital o dificultades motrices.

Recomendaciones:

- Botones de mínimo 56px de alto.
- Separación amplia entre botones.
- Texto mínimo de 16px.
- Contraste alto.
- Íconos simples.
- Evitar gestos complejos.
- Evitar menús ocultos.
- Evitar acciones importantes en esquinas pequeñas.
- Confirmaciones visuales y textuales.
- No depender solo del color para comunicar estado.

---

## 23. Aplicación al SEM Digital

La interfaz debe sentirse como parte del ecosistema municipal, pero más simple.

La app del permisionario debe priorizar:

- Rapidez.
- Claridad.
- Botones grandes.
- Poca carga cognitiva.
- Registro del pago en pocos segundos.

El panel municipal debe priorizar:

- Trazabilidad.
- Recaudación.
- Control administrativo.
- Consulta por patente.
- Visualización de pagos.

---

## 24. Do / Don't

### Hacer

- Usar azul institucional como color principal.
- Usar blanco para limpieza visual.
- Usar verde solo para confirmaciones.
- Usar botones grandes.
- Usar textos cortos.
- Mostrar la patente en grande.
- Mostrar vencimiento en grande.
- Mantener el logo visible en pantallas clave.
- Usar cards redondeadas.
- Usar íconos simples.

---

### No hacer

- No usar interfaces oscuras complejas.
- No sobrecargar con datos.
- No usar lenguaje técnico.
- No pedir datos del conductor.
- No usar tablas en la app del permisionario.
- No esconder la acción principal.
- No poner botones peligrosos cerca del botón principal.
- No hacer que el conductor dependa de una app.
- No romper la línea azul/blanco institucional.

---

## 25. Resumen visual

La interfaz debe verse como:

```text
Azul + blanco
Botones grandes
Cards redondeadas
Tipografía geométrica
Íconos simples
Confirmaciones verdes
Lenguaje directo
Sistema institucional, moderno y accesible
```
